1.   Introduction

This analysis document identifies the requirements of the Calvary Chapel of Austin Internet Ministry for content management of the church web site. The goal is to enable ministry leaders, servants, and church staff to create and publish new content to the web site without requiring technical skills for all but the most difficult of tasks. The document will also provide CMS resources, possible CMS solutions, and a requirements matrix of CMS solutions that met at least 50% of the identified requirements.

2.   Who Should Read This Document

This document is intended for the pastoral staff and internet ministry servants, as well as anyone interested in the analysis phase for the CMS solution.

3.   Content Management System Overview

A Content Management System (CMS) is a tool that enables a variety of (centralized) technical and (de-centralized) non technical staff to create, edit, manage and finally publish a variety of content (such as text, graphics, video etc), whilst being constrained by a centralized set of rules, process and workflows that ensure a coherent, validated website appearance.

 

Unfortunately, CMS has been used to identify any number of things that manage content or even just web pages. This document assumes that a CMS matches the definition above, but sub-categorizes them in the following manner: CMS frameworks and CMS solutions. CMS frameworks focus primarily on the server-side functionality necessary to create a CMS solution, but more than likely only offer a skeleton solution or no solution at all. Thus, programming is required to make a framework become a solution. A CMS solution is considered a fully-functional CMS that offers both the server-side functionality and a user interface for content management. In addition, a solution may offer either programmatic methods of customizing the CMS to suit one’s needs, or may offer a user interface for customization.

 

Other software solutions sometimes classified as a CMS include portal engines (PHPNuke, Slashcode, etc.), website management tools (Microsoft Frontpage), and Knowledge Management Systems (Wiki). These items, though they manage content, are not considered a full-blown CMS and are not included in this analysis document.

 

Note: There were found at Sourceforge (a hosting site for software projects) over 350 projects labeled “CMS” but either have not released any code or are not really a CMS after all.

 

4.   Content Management System Requirements

The following section outlines the requirements identified for the CA Internet Ministry’s content management solution. Items marked with a must are required features of the CMS solution to be considered for selection. Items marked should are a high priority, but are not considered an essential requirement of the CMS solution. Items marked may are considered value-add only.

4.1.                    Functional

The functional section identifies requirements that are required functions of the CMS solution.

4.1.1.   Web Content Management

The system must be designed to manage web-based content.

4.1.2.   Other Content Management

The system should allow other types of content to be managed, such as Word documents, PowerPoint presentations, and PDF files.

4.1.3.   Content Import/Export

The system should allow content to be imported and exported using multiple formats, such as plain text, PDF, and Word/Excel.

4.1.4.   Current Content Import

The system should store data in a manner that would allow content from the existing web site, in XML format, to be transformed to its internal format or pushed in via an API.

4.1.5.   Content Editing

The system must have a method of editing content via the web using either simple HTML or HTML-like markup capabilities.

4.1.6.   Content Security

The system must be able to assign read/write permissions to content to prevent unwanted editing of content by other ministries and staff.

4.1.7.   Workflow-based Publishing

The system must offer new content to be reviewed and edited before being published to the public.

4.1.8.   Automatic Publishing

The system should offer an option to allow content to be automatically published, to enable weekly changes to skip the review process (events, audio catalog, etc).

4.1.9.   Versioning

The system must offer to version historical versions of content, allowing for editors to approve changes to existing documents.

4.1.10.                       Image Support

The system should offer an option to allow content creators/editors to upload images to the system and link images into the content.

4.1.11.                       Writers/Editors User Account

The system must offer signup for creating a user account for writers and editors and allow customizations to be stored and available on the next login.

4.1.12.                       Online Help

The system should offer online help to content writers and editors, to reduce support overhead on the Internet Ministry.

4.1.13.                       Page Layout

The system should offer more than one page layout style, to enable text and images to be offered in a variety of pleasing formats.

4.1.14.                       Static Page Generation

The system should offer capabilities to export published content as static HTML rather than dynamic, DB generated pages to enable easier replication and less runtime dependence on the content management system.

4.1.15.                       Staging/Preview Capability

The system should offer a method of previewing new/edited pages to determine the overall impact of the addition/change.

4.1.16.                       Search

The system should offer a search capability so that visitors may locate articles quickly.

4.1.17.                       Content Filtering

The system should offer a feature to scan for profanity if features are open to the public, such as discussion forums.

4.1.18.                       Search Engine Keywords

The system should allow writers and editors to specify keywords for the content and use these keywords to generate proper HTML for search engines to properly index the site.

4.1.19.                       Pluggable Extensions

The system should offer an option to extend the functionality of the CMS via extensions (e.g. portlets or module-based concepts)

4.1.20.                       Surveys

The system should offer a method of designing surveys for visitors. The system must offer reporting capabilities for the surveys if the survey feature is supported.

4.1.21.                       Shopping Cart

The system should offer a shopping cart for visitors to purchase books, tapes, CDs, and event signups.

4.1.22.                       E-Commerce

The system must offer an online payment system or the capability if the shopping cart feature is supported, to enable visitors to pay for items in their shopping cart.

4.1.23.                       E-Commerce User Account

The system must offer user accounts and allow customizations to be stored and available on the next login if the e-commerce feature is supported

4.1.24.                       Private Areas

The system should offer a private area that is accessible via their user account, controlled by a role-based access. This enables staff members to store private information available only to other staff rather than all staff and visitors.

4.1.25.                       Discussion Forums

The system should offer customizable discussion forms that offer moderated as well as non-moderated forums.

4.1.26.                       Look-n-Feel

The system should offer pluggable “skins” or offer documentation as to the steps required to replace the current look-and-feel. This may be satisfied by a simple templating or master template concept.

4.1.27.                       Status Notification

The system must send an email with the new/edited content that was submitted to the editorial staff for review.

4.1.28.                       Task/Milestone Management

The system should offer a method of managing progress for tasks/milestones associated with content generation.

4.1.29.                       Reporting

The system should offer reporting capabilities to determine usage statistics, as well as editing history (what pages were changed within a specified time period, who changed them, bad links, etc.).

4.1.30.                       Report Export

The system should offer the ability to export reporting data to an external format for weekly/monthly status reports (e.g. PDF, XML, Excel, and Word).

4.1.31.                       Timed Publishing

The system should offer a feature to set the time/date when an article may be published to the public after being edited.

4.2.                    Non-Functional

Non-functional requirements are items identified as requirements but are not considered features within a CMS.

4.2.1.   Operating System

The system must support the Linux operating system, and should support the Windows 2000/XP operating system for extensions development by the ministry.

4.2.2.   Programming Language

The system should be written in Java due to the church’s availability of Java software developers.

4.2.3.   Database Support

The system should support a database that is free and available on the Linux operating system.

4.2.4.   Documentation

The system should offer some level of documentation for the setup, customization, and administration of the site.

It must offer documentation if online help is not offered

4.2.5.   Customization

The system should require minimal customization work using programming to ensure content can be managed. Programmatic customization may be required to extend the capabilities not supported by the system but are optional requirements as defined by this document.

4.2.6.   Usability

The system must be easy to use for content writers to enable all servants to learn how to create/edit content using the system with minimal training and support.

4.2.7.   Cost

The system must have a cost of $500 or under. Preferably, the system should be open source so that changes can be made to the system without additional monetary cost.

 

5.   Conclusions

After evaluating the solutions with the best potential for satisfying the Calvary Austin Internet Ministry requirements, it was discovered that most solutions on the market either: 1) cost too much, 2) are composed of too many technical solutions that do not solve a complete problem set, and/or 3) didn’t truly understand the purpose of a CMS solution after claiming to be a CMS.

 

It is with careful consideration and prayer that the following decisions were made:

 

  1. A solution will be built internally to solve the ministry’s content management needs
  2. There are a number of frameworks available to assist in this custom solution
  3. These solutions include Slide (WebDAV-enabled content management framework and user management system) and Cocoon (an XML/XSL framework) that requires a J2EE server environment.
  4. The church currently has a number of Java developers in the local body that can assist the ministry in building this solution.
  5. We will use the existing XML + XSL infrastructure to separate content and layout
  6. Microsoft’s emerging XDocs solution and move to a pure XML-based Microsoft Office 10 indicates XML as a content solution will become more mainstream – even more than HTML itself

 

With these decisions in place, the next document to be constructed and reviewed at this time is a software architecture document (SAD) to outline the software solution to meet the requirements documented above.

8.   Evaluation Results

 

The following section outlines the findings from the CMS solutions evaluated.

 

8.1.                    OpenCMS

Pros: Java engine powerful, with module extension capabilities; open source; interesting but high learning curve to understand the custom Java content generation; documentation is good, with plenty of tutorials

Cons: Requires an ActiveX control and Windows OS to properly edit; too loose during the edit; no online help;  very painful process to create and edit a document; unusual UI

Summary: Not sure of when a pure Java solution will be out, so non-Windows users would have to code HTML in the code editor or use a PC; Powerful, but will take about 4-6 weeks of setup and work to get it ready for the ministry and still won’t offer all that we need; may want to contact mailing list and let them know/see if anyone has a better solution for the UI
